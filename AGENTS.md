# 项目协作规范

本文件是仓库内人类与 AI 协作者共同遵循的主规范，适用于整个目录树。

## 1. 范围

- 项目只构建和输出 ETF 池。
- 不实现交易信号、仓位、组合优化、收益评估或下单功能。
- 核心代码放在 `src/etf_pool/`；脚本只作为薄入口。
- 默认数据源为 Tushare，数据源差异由 `data` 层隔离。

## 2. 环境

- 使用 Conda 环境 `quant` 和 Python 3.10。
- `pyproject.toml` 是 Python 包依赖的权威声明，`environment.yml` 用于复现 Conda 环境。
- 新增运行时依赖时必须同步更新两者，并说明必要性。

## 3. 数据与密钥

- 禁止提交 `.env`、Token、Cookie、账号或代理凭据。
- 禁止提交原始行情、大型中间表和生成的 ETF 池；Git 只保留结构与数据契约。
- `data/raw/` 保存不可变原始快照；修正数据时写新快照，不静默覆盖历史。
- 日行情以 `(ts_code, trade_date)` 为自然键，写入前检查重复键。
- 每个 ETF 池结果必须记录 `as_of_date`、数据来源、抓取时间、配置和代码版本。

## 4. ETF 池规则

- 筛选条件必须可解释，入池和排除都要保留原因。
- 上市天数、行情条数、成交额等指标必须明确单位、窗口和缺失值处理。
- 不用当前 ETF 列表覆盖旧快照；每次运行保留当时的数据边界。
- 跟踪同一指数的产品去重规则必须固定且可审计。
- 任何新规则都要先更新配置或数据契约，再实现代码和测试。

## 5. 代码规范

- Python 使用类型标注，公共函数提供简洁 docstring。
- 路径使用 `pathlib.Path`，结构化配置使用标准解析器，表格使用 pandas API。
- 不在模块导入时请求网络、读取密钥或写文件。
- 外部 API 只出现在 `data` 层，筛选逻辑只接受明确的表格输入。
- 错误必须可操作，不打印 Token，不吞掉异常。
- 变更保持聚焦，不覆盖用户本地数据和无关改动。

## 6. 测试与交付

- 筛选规则使用小型确定性表格测试边界值、缺失值和重复键。
- 默认测试不依赖网络或真实 Token；API 测试使用 fixture 或 mock。
- 缺陷修复必须增加回归测试。
- 提交前运行 `pytest`；共享模块变更还应运行 `ruff check .` 和 `mypy src`。
- 字段或存储变化同步更新 `docs/data-contracts.md`，职责变化同步更新 `docs/architecture.md`。

## 7. Git

- 提交前使用 `git status --ignored` 检查 `.env` 和本地数据状态。
- 分支建议使用 `feature/`、`fix/`、`data/` 前缀。
- 提交必须遵循 `type(scope): subject` 格式，具体类型和示例见 `docs/git-conventions.md`。
- 首行最多 72 个字符；scope 使用小写英文且仅描述主要影响模块。
- 克隆仓库后运行 `make git-setup`，启用提交模板和格式校验钩子。
- 不重写共享分支历史，不用破坏性命令处理不属于自己的改动。
